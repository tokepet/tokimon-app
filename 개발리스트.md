## 1. MVP 정의

**FE와 BE는 MVP 안에서 통합되지 않는다.** 백엔드는 SQLite에 토큰을 쌓고, 프론트엔드는 펫 선택만 처리한다. 두 결과가 같은 머신 위에서 동시에 잘 도는 것까지가 MVP의 통합 검증이다. 펫이 토큰을 먹어 자라는 흐름은 6/3 이후 첫 번째 작업으로 분리한다.

| 트랙 | MVP 범위 (포함) | 명시적 비MVP |
|---|---|---|
| **Frontend** | 떠다니는 윈도우 + 첫 실행 시 3종 스타터 펫 중 1마리 선택 + 선택 펫 idle 표시 + 선택 영속 | 진화 애니메이션 · 피딩 UI · 가차 · HUD · 인벤토리 |
| **Backend** (Python sidecar) | Anthropic API · OpenAI API · Claude Code CLI의 토큰 수집 + SQLite 저장 + 디둡 + TokiPoint 단순 합 계산 + FastAPI + httpx + watchdog + PyInstaller 번들 | BE→FE 실시간 push · Gemini · 클라우드 동기화 · 수동 입력 |
| **Design** | 3종 스타터 펫 Lottie idle 애니메이션 + 펫 선택 카드 디자인 | 진화 단계 애니메이션 · 상태(Excited·Sleepy 등) · 가차 연출 |

---

## 2. 팀 구조와 병렬성

세 트랙이 거의 독립적으로 굴러간다. 백엔드와 프론트엔드 사이에 MVP 안에서 공유할 API 계약이 없기 때문이다.

```
Backend 트랙 (Python sidecar)        →  SQLite에 토큰 기록
                                        (PyInstaller로 단일 바이너리)
  · 5/29까지 FE와 의존성 없음

Frontend 트랙 (Tauri Rust 셸 + React) →  데스크탑 떠다니는 펫 (선택 1마리)
  · 디자이너 Lottie 자산 의존 (5/27 데드라인)
  · Rust 코드는 ~150–270줄로 최소화 (셸 · 사이드카 spawn · 윈도우 제어)
  · BE와 통합은 Post-MVP

Design 트랙 (Lottie)                  →  [5/27 자산 납품] → FE Sprint 2에 합류
```

PM은 양 트랙을 별도 standup으로 운영해도 무방하다. 통합은 6/2–6/3 이틀에 몰아서 잡힌다.
