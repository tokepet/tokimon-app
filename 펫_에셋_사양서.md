# TokiMon 펫 에셋 사양서

이 문서는 TokiMon 프로젝트에 펫을 추가할 때 디자이너·개발자가 공통으로 따르는 규약입니다. 펫은 단일 인터페이스를 구현하는 **객체**로 다뤄지며, 같은 사양에 맞춰 에셋과 manifest를 제공하면 사용처(StarterPicker, Wanderer 등) 코드 변경 없이 자동으로 통합됩니다.

---

## 1. Pet 객체 모델

### 1.1 책임

TokiMon의 펫 객체는 다음 다섯 가지 책임을 가집니다.

| 책임 | 설명 |
|---|---|
| **Identity** | 자신의 식별자·이름·설명을 가진다 |
| **Mood** | 사용 패턴에 따라 5가지 상태를 자동 전환한다 |
| **Events** | 사용자 입력(클릭/드래그 등)에 반응한다 |
| **Movement** | 화면 내에서 자율적으로 이동한다 |
| **Evolution** | 진화 단계를 가지며 단계 전환 시 변신한다 |

각 책임은 별도의 스프라이트 시트와 메타(`manifest.json`)로 구현되며, 시스템은 펫 객체를 인터페이스로만 다룹니다(구현 세부는 캡슐화).

### 1.2 인터페이스 (TypeScript)

```ts
type Mood = "idle" | "excited" | "sleepy" | "hungry" | "curious";
type MovementKey = "walk" | "walk_left" | "walk_right";
type EventKey = "click" | "drag_hold" | "drag_release" | "double_click_react";
type EvolutionKey = "egg_idle" | "evolve" | "adult_idle";
type PolishKey = "blink" | "look_around" | "stretch" | "yawn" | "wave";

type PetState = Mood | MovementKey | EventKey | EvolutionKey | PolishKey;

interface PetSpriteDef {
  src: string;
  frameCount: number;
  frameWidth: number;
  frameHeight: number;
  fps: number;
}

interface PetDef {
  id: string;            // kebab-case
  name: string;          // UI 표시명
  description: string;
  displaySize: number;   // 화면 표시 한 변(px)
  states: Partial<Record<PetState, PetSpriteDef>>;
}
```

모든 펫은 `PetDef`를 구현하며, 시스템은 `id` 기반으로 펫을 조회·활용합니다. 새 펫 추가는 `PETS` 배열에 한 항목 추가하는 것으로 끝나며, 사용처 컴포넌트는 수정하지 않습니다.

### 1.3 Identity (속성)

| 키 | 타입 | 설명 |
|---|---|---|
| `id` | string | 영문 소문자 + 하이픈(kebab-case). 디렉토리명·코드 키 |
| `name` | string | UI 표시 이름. 한글 가능 |
| `description` | string | 한 줄 설명 |
| `displaySize` | number | 화면 표시 크기(px). 정사각 기준 |
| `canvasSize` | number | 원본 에셋 한 변(px). 256 또는 512 권장 (manifest 전용) |

### 1.4 Mood (상태)

펫은 사용 패턴에 따라 다음 5가지 상태를 자동 전환합니다. 사용자가 직접 선택하지 않습니다.

| Mood | 트리거 | 동작 |
|---|---|---|
| `idle` | 기본 상태 | 호흡 애니메이션 (1~2px 미세 진폭) |
| `excited` | 1분 안에 대량 토큰 유입 | 활기찬 모션 + 파티클 |
| `sleepy` | 24시간 이상 미사용 | 졸린 표정 + Z + 슬로 모션 |
| `hungry` | 7일 이상 미사용 | 시들한 표정 (회복형, 사망 없음) |
| `curious` | 사용자 클릭/드래그 | 시선 추적 + 짧은 반응 모션 |

### 1.5 Events (인터랙션)

| 이벤트 | 사양 |
|---|---|
| `click` | 시선 향함 + 짧은 모션(점프/윙크) 1회 후 idle 복귀 |
| `drag_hold` | 끌려가는 자세 (드래그 시간 동안 유지) |
| `drag_release` | 떨어지는 모션 + 착지 (선택) |
| `double_click_react` | HUD 토글 시 짧은 반응(까딱). 선택 |

### 1.6 Movement (이동)

| 동작 | 사양 |
|---|---|
| `walk` | 기본 이동 자세 (방향 무관 단일 시트) |
| `walk_left` | 좌향 자세. `walk` 미러링 허용 |
| `walk_right` | 우향 자세. `walk` 미러링 허용 |

### 1.7 Evolution (라이프사이클)

종(species)마다 3단계 트리: 알(egg) → 어린이(child) → 성체(adult). 일부 희귀 종은 4단계까지.

| 단계 | 사양 |
|---|---|
| `egg_idle` | 알 형태 idle (살짝 흔들림) |
| (child) | 기본 펫. 모든 Mood/Event 기본 구현 |
| `evolve` | 단계 전환 애니메이션 (1~2초, 8~12프레임) |
| `adult_idle` 등 | 성체 단계의 idle/walk/sleepy 등 (모든 child 동작 복제) |

### 1.8 Polish (보조 디테일)

| 동작 | 설명 |
|---|---|
| `blink` | 눈 깜빡임 (idle 중 5~10초마다, 0.1초) |
| `look_around` | 좌우 둘러보기 |
| `stretch` / `yawn` | 기지개·하품 |
| `wave` | 첫 실행 환영 모션 |

## 2. 에셋 사양

### 2.1 공통 기술 요구

| 항목 | 요구 |
|---|---|
| 포맷 | PNG 또는 WebP, **RGBA(알파 채널 필수)** |
| 캔버스 | 정사각, 한 변 **256 또는 512** (표시 크기의 정수 배율) |
| 배경 | **완전 투명** (체커보드가 픽셀로 박혀선 안 됨) |
| 색공간 | sRGB |
| 앵커 정렬 | 모든 프레임에서 발(앵커) 위치 동일 |
| 익스포트 형식 | 가로 스프라이트 시트(권장) 또는 프레임 시퀀스(폴더 + 001/002/…) |
| 자릿수 | 시퀀스의 경우 통일된 자릿수(예: 001, 002, …, 024) |
| JPEG | **금지** (알파 채널 없음) |

### 2.2 동작별 권장 스펙

```
[필수]
- idle           2~4프레임 · 2~3초 루프 · 호흡 1~2px 오르내림

[P0]
- walk_left      2~4프레임 · 0.5초 루프
- walk_right     2~4프레임 · 0.5초 루프 (walk_left 미러링 허용)
- click          3~5프레임 · 0.3~0.5초 1회 (점프 또는 윙크)
- drag_hold      1~2프레임 · 정적 (매달림/늘어짐 자세)

[P1]
- sleepy         2~3프레임 · 3~4초 루프 · Z 효과
- excited        4~6프레임 · 1초 루프 (파티클 별도 또는 합성)

[P2]
- hungry         2~3프레임 · 3초 루프 · 시들한 표정
- curious        4~6프레임 · 1초 · 시선 추적 모션

[P3 (진화 도입 시)]
- egg_idle       2~3프레임 · 2초 루프 · 좌우 살짝 흔들림
- evolve         8~12프레임 · 1~2초 1회 (단계 전환)
- adult_*        성체 단계의 idle/walk/sleepy 등
```

### 2.3 우선순위 매트릭스

| 동작 | 필수 | P0 | P1 | P2+ |
|---|:---:|:---:|:---:|:---:|
| idle | ✅ | | | |
| walk_left / walk_right | | ✅ | | |
| click | | ✅ | | |
| drag_hold | | ✅ | | |
| sleepy | | | ✅ | |
| excited | | | ✅ | |
| hungry | | | | ✅ |
| curious | | | | ✅ |
| evolve / egg / adult | | | | ✅ |
| blink / look / stretch | | | | ✅ (폴리시) |

## 3. 디렉토리·파일 구조

```
tokimon-app/
└── public/pets/<id>/
    ├── idle.png           [필수]
    ├── walk.png           [P0]
    ├── walk_left.png      [P0, 미러링 허용]
    ├── walk_right.png     [P0, 미러링 허용]
    ├── click.png          [P0]
    ├── drag_hold.png      [P0]
    ├── sleepy.png         [P1]
    ├── excited.png        [P1, eat.png alias 가능]
    ├── hungry.png         [P2, sad.png alias 가능]
    ├── curious.png        [P2]
    ├── manifest.json      [필수]
    ├── frames/            [선택, 시퀀스 작업본 보존]
    └── raw/               [선택, 원본 번호 파일 보존]
```

> `id`는 영문 소문자 + 하이픈(kebab-case)만 사용.

## 4. manifest.json 스키마

각 펫 폴더 루트에 `manifest.json` 1개. **이 파일이 펫 객체의 단일 진실의 소스(single source of truth)** 입니다. 코드는 이 파일을 신뢰합니다.

### 4.1 필수 필드

```json
{
  "id": "<kebab-case-id>",
  "name": "<UI 표시 이름>",
  "description": "<한 줄 설명>",
  "displaySize": 80,
  "canvasSize": 256,
  "sprites": {
    "<state>": {
      "file": "<파일명.png>",
      "frames": <프레임 수>,
      "fps": <초당 프레임>,
      "loop": <true | false>
    }
  }
}
```

### 4.2 옵셔널 필드

| 필드 | 설명 |
|---|---|
| `sprites[state].derivedFrom` | 다른 상태에서 파생됐음을 기록 (예: `hungry` → `sad`) |
| `sprites[state].mirrored` | 좌우 반전 여부 (예: `walk_left` → `walk` 미러링) |
| `sourceMapping` | 원본 파일명 → 정리된 파일명 매핑(작업 흔적 보존용) |

### 4.3 상태 키 분류

| 분류 | 키 |
|---|---|
| Mood | `idle`, `excited`, `sleepy`, `hungry`, `curious` |
| Movement | `walk`, `walk_left`, `walk_right` |
| Event | `click`, `drag_hold`, `drag_release`, `double_click_react` |
| Evolution | `egg_idle`, `evolve`, `adult_idle`, `adult_walk` 등 |
| Polish | `blink`, `look_around`, `stretch`, `yawn`, `wave` |

## 6. 코드 통합 흐름

디자이너가 위 규약대로 폴더와 manifest를 제공하면 통합은 다음 3단계.

### 6.1 에셋 배치
받은 `<id>/` 폴더를 `tokimon-app/public/pets/`에 그대로 배치. 같은 파일명 덮어쓰기 시 webview 캐시 갱신(⌘R) 필요.

### 6.2 카탈로그 등록
`tokimon-app/src/domain/petCatalog.ts`의 `PETS` 배열에 `PetDef` 인스턴스 한 개 추가. manifest의 `sprites`를 그대로 매핑.

### 6.3 검증
1. `npx tsc --noEmit` — 타입 통과
2. `npm run build` — 번들 통과
3. `npm run tauri:dev` — 데스크탑 윈도우에서 시각 검증

> 펫 수가 늘면 manifest.json → petCatalog.ts 자동 변환 스크립트 도입 권장(현재는 수동 매핑).

## 7. 품질 검증 기준

| 검증 | 도구/방법 |
|---|---|
| 알파 채널 보존 | `sips -g hasAlpha <file>` → `yes` |
| 정사각 프레임 | `pixelWidth == pixelHeight × frameCount` |
| 정수 배율 권장 | `canvasSize % displaySize == 0` |
| 앵커 정렬 | 프레임 토글 시 위치 흔들림 없음 |
| 호흡 진폭 | idle 두 프레임 비교 시 1~2px 차이 (너무 미세하면 박제 느낌) |
| 알파 가장자리 | 자동 투명화 아티팩트 없음 (디자이너 RGBA 원본 권장) |

## 8. 변경 이력 관리

- 펫 에셋 변경 시 `manifest.json`도 함께 갱신
- 상태 추가/삭제, 캔버스 크기 변경 같은 큰 변경은 PR 또는 디자이너 변경노트로 공유
- 동일 파일명 덮어쓰기 시 webview 캐시 갱신 (⌘R 또는 cache-busting query)
- 본격 배포 빌드에선 `<file>?v=<hash>` 같은 cache-busting 자동화 권장

## 9. 객체지향 설계 원칙 요약

| 원칙 | 적용 |
|---|---|
| **인터페이스 일관성** | 모든 펫은 `PetDef`를 구현. 사용처 코드는 펫 종류를 알 필요 없음 |
| **캡슐화** | 펫의 시각·동작 디테일은 `manifest.json` 내부에 갇히고, 외부로는 `PetDef` 인터페이스만 노출 |
| **단일 진실의 소스** | 펫 메타는 `manifest.json` 1곳에서만 관리. 코드는 이를 신뢰 |
| **확장 개방·수정 폐쇄** | 새 펫 추가 = `PETS` 배열에 인스턴스 1개 추가. 기존 컴포넌트 수정 불필요 |
| **다형성** | 같은 `<PetView pet={...} state={...} />`로 어떤 펫도 동일하게 표시 |
