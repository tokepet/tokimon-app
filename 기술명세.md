## 1. 기술 스택

### 1.0 언어 사용 정책 — "Rust 최소화"

팀의 강점은 Python이고, MVP 마감은 13 working days라 빠르다. 그래서 본 프로젝트는 **회피 불가능한 곳에서만 Rust를 쓰고, 그 외 모든 백엔드 로직은 Python으로 작성**한다는 정책을 명시한다.

이 정책의 부수 결과는 다음과 같다.

- 유휴 메모리 목표가 50MB에서 **80MB**로 완화된다 (§5 참조).
- 인스톨러 크기가 Python 사이드카 분량만큼 +15MB 정도 늘어난다.
- 팀 학습 비용이 0이며, 백엔드 개발 속도가 1.5~2배 빨라진다.

각 Rust 사용처는 다음 1.1 표에 **왜 회피가 불가능한지**를 명시한다. 새 기능이 Rust를 요구한다면 그 이유가 표의 어떤 항목과 같은 결인지부터 확인한다.

### 1.1 Rust로 작성하는 부분 (회피 불가, 코드량 최소화)

| 컴포넌트 | 회피 불가능한 이유 | 추정 코드량 |
|---|---|---|
| **Tauri 앱 셸** (윈도우 생성 · 시스템 트레이 · 메뉴) | Tauri 2.x 프레임워크 자체가 Rust로 작성됨. 앱의 진입점(`fn main()`)이 Rust 함수다. webview 호스팅, 윈도우 옵션(투명/프레임리스/Always-on-top), 시스템 트레이는 Tauri의 Rust API를 거쳐야만 호출 가능 | 100–200 줄 |
| **사이드카 spawn 설정** | Tauri가 Python 수집기 바이너리를 자식 프로세스로 띄우려면 `tauri.conf.json`의 `bundle.externalBin`과 Rust의 `Command::new_sidecar()`가 필요. 이게 사이드카 패턴의 정의다 | ~20 줄 |
| **webview ↔ OS 윈도우 제어 명령** | React가 "윈도우 드래그 시작" · "Always-on-top 토글" 같은 OS 기능을 호출하려면 Tauri command(Rust 함수)로 노출해야 한다. webview JS만으로는 OS를 직접 제어할 수 없다 | ~50 줄 |

**합계 Rust 코드: 약 150–270 줄.** 나머지 백엔드는 전부 Python이다.

### 1.2 Python으로 작성하는 부분 (백엔드 전부)

| 컴포넌트 | 라이브러리 | 선택 이유 |
|---|---|---|
| HTTP 프록시 서버 | **FastAPI + uvicorn** | 팀 숙련도 ↑. async + SSE streaming 우수. 더 가볍게 가려면 **aiohttp** 옵션 (메모리 −10MB) |
| API forwarding | **httpx (async)** | `aiter_lines()` 한 줄로 SSE 청크 파싱 |
| 어댑터 추상화 | **`abc.ABC` + `@abstractmethod`** | Python idiom, Rust trait의 직접 대응 |
| 로컬 저장소 | **`sqlite3` (stdlib)** 또는 **aiosqlite** | 추가 의존성 0. 동기·비동기 모두 가능 |
| 로그 파일 watch | **watchdog** | 크로스플랫폼 파일 시스템 이벤트. Claude Code JSONL tail에 사용 |
| TokiPoint 계산 | (자체 함수) | 단순 합산이라 의존성 불필요 |
| CLI 진입점 (디버깅용) | **Typer** 또는 Click | 개발자가 수집기를 단독으로 실행/검증 |
| 번들링 | **PyInstaller `--onefile`** | 단일 ~20MB 실행 바이너리 생성. Tauri의 `externalBin`으로 첨부되어 앱 시작 시 자동 spawn |

### 1.3 Frontend (Tauri webview 내부)

| 컴포넌트 | 선택 |
|---|---|
| UI 프레임워크 | **React + TypeScript** |
| 라우팅 | **React Router** (선택 화면 ↔ 펫 화면) |
| 펫 애니메이션 | **lottie-react** (벡터 SVG 기반 Lottie 렌더) |
| 사이드카 통신 | **localhost HTTP** — Python sidecar의 `http://127.0.0.1:39xx`를 React가 직접 fetch. Rust IPC를 우회해서 Rust 코드 추가 작성을 회피한다 |

### 1.4 인프라

| 항목 | 선택 |
|---|---|
| 클라우드 동기화 (Post-MVP) | Supabase 또는 Cloudflare D1 |
| 패키징 | Tauri Bundler (`.dmg`, `.msi`) |
| 자동 업데이트 (Post-MVP) | Tauri Updater (ed25519 서명) |

## 2. 시스템 아키텍처

전체 구성을 한 장에 보여주는 다이어그램이다. **굵은 박스가 Rust 영역**이고, 그 외는 모두 Python 또는 TypeScript다. 점선은 프로세스 경계를 의미한다.

### 통신 경로를 한 줄씩 설명하면

- **사용자 외부 SDK → Python 프록시**: 환경변수 redirect로 사용자의 SDK 호출이 localhost로 향한다.
- **React UI → Python 프록시**: localhost HTTP fetch. Rust IPC를 거치지 않아서 Rust 코드를 추가로 작성할 필요가 없다.
- **Tauri 셸 ↔ Python 사이드카**: 앱 시작 시 spawn, 종료 시 kill. Rust 진입점에서만 일어난다.
- **Tauri 셸 ↔ React webview**: Tauri의 `invoke()`. 윈도우 제어 같은 OS 기능을 호출할 때만 쓴다.

## 3. 토큰 수집 — 멀티 프로바이더

여러 AI 서비스에서 토큰을 모으는데, 서비스마다 기술적 제약이 달라 단일 수집 방식으로는 커버하기 어렵다. 그래서 4가지 수집 방식을 병행한다.

### 3.1 수집 전략 4가지

| 방식 | 적용 대상 | 장점 | 한계 |
|---|---|---|---|
| **A. 로컬 프록시** | API SDK를 쓰는 서비스 (Anthropic · OpenAI · Gemini · OpenRouter 등) | 실시간, 정확, SDK 수정 불필요 | 사용자가 `*_BASE_URL` 환경변수를 설정해야 함 |
| **B. 로그·트랜스크립트 watch** | CLI 도구 (Claude Code · Gemini CLI 등) | 환경변수 설정 불필요, 백그라운드 자동 | 파일 포맷 변경에 취약, 거의 실시간(수 초 지연) |
| **C. Admin/Usage API polling** | OpenAI Usage API · Anthropic Admin API | 사용자 환경에 비침습, 모든 호출 캡처 | Admin Key 필요, 지연(수 분~수 시간) |
| **D. 수동 입력 / Import** | Cursor · Copilot처럼 후킹이 어려운 도구 | 어떤 서비스든 가능 | 사용자 입력 부담 |

### 3.2 v1에서 지원하는 프로바이더

| 프로바이더 | 수집 방식 | 추출 필드 | 우선순위 |
|---|---|---|---|
| Anthropic Claude API | A. 프록시 (`ANTHROPIC_BASE_URL`) | input · output · cache_creation · cache_read | **P0 (MVP)** |
| Claude Code CLI | B. JSONL Watch (`~/.claude/projects/`) | 동일 (transcript 안에 usage 포함) | **P0 (MVP)** |
| OpenAI API (GPT) | A. 프록시 (`OPENAI_BASE_URL`) | prompt · completion · prompt_tokens_details.cached | **P0 (MVP)** |
| OpenRouter · Together · Groq · Fireworks | A. 프록시 (OpenAI 호환 스키마) | OpenAI 어댑터 재사용 | **P0 (MVP, 자동 커버)** |
| Ollama · LM Studio (로컬 LLM) | A. 프록시 (OpenAI 호환) | OpenAI 어댑터 재사용 | **P0 (MVP, 동일)** |
| Google Gemini API | A. 프록시 (커스텀 endpoint) | input · output | P1 (Phase 1.5) |
| Cursor · GitHub Copilot | D. 수동 입력 또는 import (CSV) | 사용자가 입력한 토큰 | P2 |

### 3.3 로컬 프록시 (방식 A) 동작 방식

사용자는 셸에 프로바이더별 `*_BASE_URL`을 `http://localhost:39xx/<provider>`로 export한다. 그러면 Python 프록시가 `path prefix`(`/anthropic/*`, `/openai/*`, `/gemini/*`)로 어떤 어댑터를 쓸지 분기한다. 어댑터는 요청을 원본 도메인으로 forward한 뒤, 응답에서 usage 필드를 추출한다. streaming(SSE)과 non-streaming을 모두 처리한다.

**API 키 비저장 원칙**: 프록시는 사용자 환경변수의 Authorization 헤더를 그대로 통과시키기만 한다. 앱 DB에 저장하지 않는다.

### 3.4 로그 Watch (방식 B) 동작 방식

Claude Code는 `~/.claude/projects/**/*.jsonl` 경로에 transcript를 기록한다. `watchdog` 라이브러리로 파일 변경을 감지하고, 신규 라인이 들어오면 `message` 이벤트의 `usage` 필드를 추출한다. 중복은 (session_id, message_id) 기반으로 dedup한다.

### 3.5 어댑터 인터페이스

새 프로바이더를 추가할 때 어댑터 클래스 하나만 작성하면 끝나도록 `abc.ABC`로 추상화했다.

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass
from typing import Optional
import httpx

@dataclass
class RawUsage:
    provider: str
    model: str
    input_tokens: int
    output_tokens: int
    cache_creation_tokens: Optional[int] = None
    cache_read_tokens: Optional[int] = None
    external_id: Optional[str] = None   # 디둡용 (예: Claude Code message_id)

class ProviderAdapter(ABC):
    @property
    @abstractmethod
    def id(self) -> str: ...
        # "anthropic" / "openai" / "gemini" / ...

    @abstractmethod
    async def forward(self, request: httpx.Request) -> httpx.Response: ...
        # 사용자 요청을 원본 API로 forward (프록시 방식 A)

    @abstractmethod
    def parse_usage(self, response_body: bytes | str) -> Optional[RawUsage]: ...
        # 응답에서 usage 필드 추출. 스트리밍은 마지막 청크에서 추출

    def to_toki_points(self, raw: RawUsage) -> float:
        # v1: 단순 합. 모든 어댑터 공통이므로 기본 구현 제공
        return float(
            raw.input_tokens
            + raw.output_tokens
            + (raw.cache_creation_tokens or 0)
            + (raw.cache_read_tokens or 0)
        )
```

Anthropic 어댑터의 실제 구현은 다음과 같이 단순하다.

```python
class AnthropicAdapter(ProviderAdapter):
    id = "anthropic"
    UPSTREAM = "https://api.anthropic.com"

    async def forward(self, request):
        # request는 사용자의 SDK가 보낸 원본. 헤더(API 키 포함)는 그대로 통과.
        async with httpx.AsyncClient() as client:
            return await client.send(request.url.replace_host(self.UPSTREAM))

    def parse_usage(self, response_body):
        data = json.loads(response_body)
        u = data.get("usage", {})
        return RawUsage(
            provider=self.id,
            model=data.get("model"),
            input_tokens=u.get("input_tokens", 0),
            output_tokens=u.get("output_tokens", 0),
            cache_creation_tokens=u.get("cache_creation_input_tokens"),
            cache_read_tokens=u.get("cache_read_input_tokens"),
            external_id=data.get("id"),
        )
```

### 3.6 토큰 정규화 — TokiPoint (v1: 단순 합)

프로바이더마다 토큰의 의미가 다르다(Anthropic의 cache_read는 비용이 1/10, OpenAI의 cached_tokens도 마찬가지). 그러나 **v1은 의도적으로 모든 토큰을 1:1로 동등 합산**한다. 단순하고 투명한 게 출시 속도를 가장 빠르게 만들기 때문이다.

게임 레이어의 단위명은 **TokiPoint(TP)**다. 이 이름을 유지하는 이유는 미래에 가중치를 도입할 때 DB 마이그레이션 없이 환산식만 갈아끼울 수 있게 추상화 층을 남겨두기 위해서다.

**v1 환산식**:

```
TokiPoint = input_tokens + output_tokens
          + (cache_creation_tokens ?? 0)
          + (cache_read_tokens ?? 0)
```

프로바이더·모델·토큰 타입과 무관하게 1 raw token = 1 TP.

이 단순함이 만드는 의도된 결과 3가지:

- **캐시(`cache_read`)를 적극 활용한 사용자** — 비용은 1/10인데 TP는 똑같이 들어와 같은 비용으로 10배 빨리 진화한다. 이건 "효율적인 AI 사용 습관에 대한 보상"으로 포지셔닝한다.
- **로컬 LLM(Ollama) 사용자** — 비용은 0인데 TP는 그대로 쌓인다. v1에서는 허용하고, telemetry로 abuse 패턴을 모니터링한 뒤 v2에서 결정한다.
- **멀티 프로바이더 사용자** — 모든 프로바이더 토큰이 동등하게 합산되어, 단일 벤더 종속을 피하라는 제품 가치와 정렬된다.

**저장 정책**: `token_events` 테이블에 **원시 토큰을 그대로 저장**한다(감사·통계·미래 재계산용). 그와 별도로 `toki_points` 컬럼에 환산 결과도 저장한다(집계 쿼리 성능). 환산 로직은 `ProviderAdapter.to_toki_points()` 메서드 하나에 격리되어 있어서, 미래에 가중치를 도입할 때 이 함수만 교체하면 된다.

**v2에서 검토할 항목** (현재는 비활성):

- 비용 기반 환산 (`tokens × per_million_price`)
- 프로바이더/모델별 가중치 외부 JSON
- 토큰 타입별 가중치 (output > input > cache 등)

## 4. 데이터 모델 (SQLite)

로컬에 두는 단일 SQLite 파일에 모든 상태를 저장한다. 스키마는 4개 테이블이다.

```sql
-- 토큰 사용 원장 (멀티 프로바이더)
CREATE TABLE token_events (
  id INTEGER PRIMARY KEY,
  ts INTEGER NOT NULL,            -- unix epoch ms
  provider TEXT NOT NULL,         -- "anthropic" / "openai" / "gemini" / "ollama" / ...
  model TEXT,                     -- claude-opus-4-7 / gpt-4o / gemini-2.5-pro 등
  input_tokens INTEGER,
  output_tokens INTEGER,
  cache_creation_tokens INTEGER,  -- 미지원 프로바이더는 NULL
  cache_read_tokens INTEGER,      -- 미지원 프로바이더는 NULL
  toki_points REAL,               -- 정규화된 TokiPoint (게임 로직용)
  collection_method TEXT,         -- "proxy" / "log_watch" / "admin_api" / "manual" / "import"
  external_id TEXT,               -- 디둡용 (예: Claude Code message_id)
  UNIQUE(provider, external_id)
);
CREATE INDEX idx_token_events_ts ON token_events(ts);
CREATE INDEX idx_token_events_provider ON token_events(provider, ts);

-- 사용자 보유 펫
CREATE TABLE pets (
  id INTEGER PRIMARY KEY,
  species_id TEXT NOT NULL,       -- "fluff_01", "data_dragon" 등
  name TEXT,                      -- 사용자 지정 닉네임
  stage INTEGER DEFAULT 0,        -- 진화 단계 (0~N)
  exp INTEGER DEFAULT 0,          -- 펫에 누적된 토큰 EXP
  hatched_at INTEGER,
  is_active INTEGER DEFAULT 0     -- 현재 데스크탑에 떠 있는 펫 (1마리만)
);

-- 종(species) 정의는 코드/JSON에 정적으로 둠 (밸런싱 핫픽스 가능)

-- 사용자 메타
CREATE TABLE user_state (
  k TEXT PRIMARY KEY,
  v TEXT
);
-- 예: cumulative_tokens, gacha_pity_count, last_sync_at, cloud_user_id

-- 이벤트 로그 (디버그/통계)
CREATE TABLE app_events (
  id INTEGER PRIMARY KEY,
  ts INTEGER, kind TEXT, payload TEXT
);
```

### 디둡(UNIQUE) 제약이 왜 중요한가

같은 호출이 두 경로로 들어올 수 있다. 예를 들어 사용자가 Claude Code를 프록시 경유로 쓰면, 프록시(방식 A)와 로그 watcher(방식 B)가 같은 메시지를 둘 다 보게 된다. `(provider, external_id)`에 UNIQUE 제약을 걸어두면 두 번째 시도가 자동으로 무시된다. 이건 통합 테스트의 핵심 검증 포인트다.

## 5. 성능 / 리소스 목표

Python 사이드카를 채택하면서 메모리와 패키지 크기 목표를 한 번 완화했다. 그 외 지표는 동일하다.

| 항목 | v1 MVP 목표 | 내역 |
|---|---|---|
| 유휴 메모리 (앱 총합) | **≤ 80MB** | Tauri 셸 ~15MB + webview ~15MB + Python 사이드카 ~50MB |
| 유휴 CPU | < 1% | 애니메이션 idle 상태 기준 |
| 프록시 forwarding overhead | < 10ms p95 | Python async 기준. API 호출 latency 100ms+ 대비 무시 가능 |
| 인터랙션 응답 | 클릭 → 반응 < 100ms | webview 내부 처리만 |
| 패키지 크기 | macOS dmg **< 30MB** / Windows msi **< 35MB** | PyInstaller로 번들된 Python 사이드카 ~20MB 포함 |
| 펫 윈도우 FPS | 60fps | Lottie 한정 |

**v2 최적화 후보**: Python 어댑터를 그대로 두고, 가장 hot한 경로(SSE 파싱)만 Rust로 재작성하면 메모리 50MB로 복귀 가능하다. 다만 MVP 시점에는 hold한다.

## 6. 보안과 프라이버시

본 프로젝트가 다루는 가장 민감한 자산은 사용자의 **AI API 키**다. 다음 4가지 원칙을 지킨다.

- **API 키 비저장**: 프록시는 사용자 환경변수의 Authorization 헤더를 통과만 시킨다. 앱 DB에는 절대 저장하지 않는다.
- **로컬 우선**: 모든 사용량 데이터는 기본적으로 로컬에 머문다. 클라우드 동기화는 명시적 옵트인이다.
- **127.0.0.1 바인딩**: 프록시는 외부 네트워크 인터페이스에 노출하지 않는다.
- **업데이트 서명**: Tauri Updater의 ed25519 서명 검증을 사용한다.

## 7. 크로스플랫폼 고려사항

macOS와 Windows에서 같은 결과를 내야 하므로, OS별 API 차이를 사전에 식별해둔다.

| 항목 | macOS | Windows |
|---|---|---|
| 투명 윈도우 | `NSWindow` decorations 제거 + `setOpaque(false)` (Tauri 옵션) | WS_EX_LAYERED + DwmEnableBlurBehind |
| Always-on-top | 기본 지원 | 기본 지원 |
| 시스템 트레이 | 메뉴바 아이콘 | 시스템 트레이 |
| 자동 시작 | LaunchAgent plist | Registry Run 키 |
| 환경변수 가이드 | `.zshrc` · `.bashrc` 안내 + 자동 추가 옵션 | `setx` + Windows Terminal 프로파일 안내 |
